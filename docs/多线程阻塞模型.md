## 1. 最原始的服务器：单线程阻塞

最简单的 TCP 服务器只能同时服务一个客户端：

```c
// tcp_server.c 中被 #if 0 注释掉的部分，展示了最原始的版本
int clientfd = accept(sockfd, ...);  // 阻塞等待第一个连接

while(1) {
    int count = recv(clientfd, buff, 256, 0);  // 阻塞等待数据
    if (count == 0) break;
    send(clientfd, buff, count, 0);
}
```

问题一目了然：`accept` 只调用了一次，第一个客户端连进来后，第二个客户端永远等在门外。

```
客户端 A ──► accept() ──► 进入服务循环
客户端 B ──► [一直在等，永远轮不到]
客户端 C ──► [一直在等，永远轮不到]
```

## 2. 进化：每连接一线程

解决思路直观：**来一个连接，开一个线程去服务它，主线程继续 accept**。

```
主线程:
  accept() ──► 新连接 A ──► 创建线程 T1 服务 A ──► 继续 accept()
  accept() ──► 新连接 B ──► 创建线程 T2 服务 B ──► 继续 accept()
  accept() ──► 新连接 C ──► 创建线程 T3 服务 C ──► 继续 accept()

线程 T1: recv(A)/send(A) 循环
线程 T2: recv(B)/send(B) 循环
线程 T3: recv(C)/send(C) 循环
```

每个线程独立阻塞在自己的 `recv` 上，互不干扰。这是最自然的并发思维：**一个任务一个工人**。

## 3. tcp_server.c 代码解析

### 3.1 线程函数

```c
void* client_thread(void* arg)
{
    int clientfd = *(int*)arg;   // 从参数取出 fd

    while(1) {
        char buff[256] = {0};
        int count = recv(clientfd, buff, 256, 0);  // 阻塞等待数据
        if (count == 0) {
            printf("client %d disconnected.\n", clientfd);
            break;                // 对端关闭连接，退出循环
        }
        send(clientfd, buff, count, 0);  // echo 回去
        printf("clientfd: %d, count: %d, buff: %s\n", clientfd, count, buff);
    }

    close(clientfd);   // 线程退出前关闭 fd
}
```

线程的逻辑很简单：循环 recv，有数据就 echo，对端断开就退出。

### 3.2 主线程：accept 循环

```c
while(1) {
    struct sockaddr_in client_addr;
    socklen_t len = sizeof(struct sockaddr);
    int clientfd = accept(sockfd, (struct sockaddr*)&client_addr, &len);
    printf("accepted\n");

    pthread_t tid;
    pthread_create(&tid, NULL, client_thread, &clientfd);  // ← 注意这里
}
```

主线程不断 `accept`，每来一个连接就 `pthread_create` 创建新线程。

## 4. 经典 Bug：线程传参的栈变量陷阱

### 4.1 Bug 在哪里

```c
// 主线程 while(1) 循环体内：
int clientfd = accept(...);          // clientfd 是栈上的局部变量
pthread_create(&tid, NULL, client_thread, &clientfd);
//                                            ↑
//                              传入的是 clientfd 的地址（栈地址）
```

### 4.2 为什么有问题

`clientfd` 是 `while(1)` 循环体内的栈变量，每次循环都在**同一个栈位置**分配：

```
时间线：

t=0: accept() 返回 clientfd=5
     pthread_create(thread1, &clientfd)
     → thread1 拿到地址 0x7fff1234，还没来得及读值

t=1: 主线程进入下一次循环
     accept() 返回 clientfd=6         ← 同一个地址 0x7fff1234 的值变了！
     pthread_create(thread2, &clientfd)

t=2: thread1 终于执行 *(int*)arg
     读到地址 0x7fff1234 的值 = 6     ← 错了！应该是 5
     thread2 也读到 6                 ← 两个线程操作同一个 fd！
```

**后果**：两个线程同时 `recv` / `send` 同一个 fd，数据错乱，fd 被 `close` 两次（double close），严重时导致 fd 被错误复用。

### 4.3 正确的修复方式

**方式一：传值而非传址（将 fd 放到堆上）**

```c
int clientfd = accept(...);
int* pfd = malloc(sizeof(int));
*pfd = clientfd;
pthread_create(&tid, NULL, client_thread, pfd);
// 线程内：int clientfd = *(int*)arg; free(arg);
```

**方式二：用指针整数强转（避免堆分配）**

```c
int clientfd = accept(...);
pthread_create(&tid, NULL, client_thread, (void*)(intptr_t)clientfd);
// 线程内：int clientfd = (int)(intptr_t)arg;
```

**方式三：传结构体（携带更多信息）**

```c
typedef struct { int fd; struct sockaddr_in addr; } ClientArg;
ClientArg* arg = malloc(sizeof(ClientArg));
arg->fd = clientfd;
pthread_create(&tid, NULL, client_thread, arg);
```

### 4.4 这个 Bug 的本质

这是**竞态条件（Race Condition）**的典型案例：

- 主线程写 `clientfd`（下一次 accept 覆盖）
- 子线程读 `*(int*)arg`
- 两个操作没有同步，谁先执行取决于 CPU 调度

这个 bug 在低并发时很难复现（因为线程创建后很快执行），但并发一高就会出现。

## 5. 多线程模型的性能瓶颈

### 5.1 线程的内存开销

每个线程默认分配 **8MB 栈空间**（Linux 默认值，可通过 `ulimit -s` 查看）：

```
100  个连接 → 100  个线程 →  800MB 栈内存
1000 个连接 → 1000 个线程 →   8GB 栈内存
10000个连接 → 10000个线程 →  80GB 栈内存（机器根本没有）
```

实际上线程数超过几千后，系统就会因内存不足或线程创建失败而崩溃。

### 5.2 上下文切换开销

CPU 在多个线程间切换时，需要保存/恢复寄存器、程序计数器、栈指针等状态，还会导致 CPU 缓存（L1/L2）失效：

```
1000 个线程，CPU 4 核
→ 每个核平均跑 250 个线程
→ 每秒进行数万次上下文切换
→ 切换本身消耗大量 CPU 时间，真正做业务的时间反而少
```

### 5.3 线程的等待浪费

每个线程大部分时间都阻塞在 `recv` 等待数据，什么都不做但占着内存和系统资源：

```
1000 个连接中，同一时刻真正有数据的可能只有 10 个
→ 990 个线程在睡眠（浪费内存）
→ 10 个线程在工作
→ 资源利用率极低
```

## 6. 为什么多线程不能解决高并发

多线程模型的根本问题是**线程与连接 1:1 绑定**：

```
连接数 = 线程数
```

而线程是重量级资源，无法无限扩展。这催生了 I/O 多路复用的思路：

```
核心转变：
  多线程：一个工人服务一个客人，来多少客人雇多少工人
  I/O多路复用：一个工人同时照看多个客人，谁有需求就服务谁
```

| 模型 | 并发方式 | 适合规模 | 典型问题 |
|------|---------|---------|---------|
| 单线程阻塞 | 无并发 | 1 个连接 | 只能服务一个客户端 |
| 每连接一线程 | 多线程 | < 1000 | 内存、上下文切换、传参 bug |
| select/poll | I/O 多路复用 | < 几千 | O(n) 扫描 |
| epoll + Reactor | 事件驱动 | 10万+ | 编程模型复杂 |

`tcp_server.c` 是演进链条的起点，它的局限性直接驱动了 `select_server.c` 的出现。
