# 网络并发模型演进

本项目包含五个文件，完整呈现了 Linux 网络服务器从最原始到高性能的演进路径。

```
tcp_server.c
    ↓  问题：线程开销大，无法支撑高并发
select_server.c
    ↓  问题：fd 上限 1024，O(n) 扫描
poll_server.c
    ↓  问题：去掉 fd 上限，但 O(n) 未解决
epoll_server.c
    ↓  问题：O(1) 就绪通知，但业务逻辑混在主循环
reactor.c
       解决：事件驱动 + 回调解耦，结构清晰可扩展
```

---

## 第一代：单进程阻塞 → 每连接一线程

**文件：`tcp_server.c`**

```c
// 主线程：来一个连接，开一个线程
while(1) {
    int clientfd = accept(sockfd, ...);
    pthread_t tid;
    pthread_create(&tid, NULL, client_thread, &clientfd);
}

// 子线程：阻塞在 recv，等数据，echo 回去
void* client_thread(void* arg) {
    int clientfd = *(int*)arg;
    while(1) {
        recv(clientfd, buff, ...);
        send(clientfd, buff, ...);
    }
}
```

**核心思想：** 一个工人服务一个客人，来多少客人雇多少工人。

**瓶颈：**
- 每个线程默认 8MB 栈，1000 个连接 = 8GB 内存
- 大量线程上下文切换，CPU 浪费在切换而非业务上
- 线程大部分时间阻塞等待，资源利用率极低

**隐藏 Bug：** `pthread_create(&tid, NULL, client_thread, &clientfd)` 传入的是栈变量的地址，下次循环 accept 会覆盖同一地址，导致多线程拿到错误的 fd。

**适合场景：** 并发 < 100，连接有耗时计算（充分利用多核），内网工具服务。

## 第二代：I/O 多路复用 —— select

**文件：`select_server.c`**

```c
fd_set rfds, rset;
FD_SET(sockfd, &rfds);

while(1) {
    rset = rfds;   // 每次调用前必须重置（select 会修改入参）
    select(maxfd+1, &rset, NULL, NULL, NULL);

    // O(n) 遍历：扫描 0~maxfd 每一个
    for(int i = 0; i <= maxfd; i++) {
        if (FD_ISSET(i, &rset)) { ... }
    }
}
```

**核心改变：** 用一个线程监听所有 fd，内核等待任意一个就绪再返回，告别"一线程一连接"。

**相比上一代解决了：** 线程开销问题，单线程可管理多个连接。

**新的瓶颈：**
- `fd_set` 是位图，硬编码上限 **1024**（`FD_SETSIZE`）
- 每次调用都要把整个 fd_set 从用户态拷贝到内核态
- 内核要扫描 **0~maxfd 全部 fd**，大量无效检查（O(n)）
- 用户态还要再遍历一次（O(n)）
- 每次调用前必须重置 `rset = rfds`

**适合场景：** 需要跨平台（Windows/macOS/Linux），并发连接数 < 几百。

## 第三代：I/O 多路复用 —— poll

**文件：`poll_server.c`**

```c
struct pollfd fds[1024];
fds[sockfd].events = POLLIN;   // events 只设置一次

while(1) {
    poll(fds, maxfd+1, -1);    // 不需要重置

    for (int i = 0; i <= maxfd; i++) {
        if (fds[i].revents & POLLIN) { ... }  // 检查 revents
    }
}
```

**核心改变：** 用 `pollfd` 结构体数组替换 `fd_set` 位图，`events`（期望）和 `revents`（实际）分离。

**相比上一代解决了：**
- fd 数量上限：去掉了 1024 的硬限制
- 不需要每次重置监听集合（events 不会被修改）
- 断开连接时用 `fds[i].fd = -1` 标记，内核自动跳过

**没有解决的问题（与 select 相同）：**
- 每次调用仍需把整个 pollfd 数组拷贝进内核
- 内核仍需 O(n) 扫描全部 pollfd
- 用户态仍需 O(n) 遍历检查 revents
- 不支持 Windows

**适合场景：** 跨 Linux/macOS/BSD，并发 < 几千，fd 值可能超过 1024。

## 第四代：I/O 多路复用 —— epoll

**文件：`epoll_server.c`**

```c
int epfd = epoll_create1(EPOLL_CLOEXEC);

// 注册一次，内核持久保存
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);

while(1) {
    int nready = epoll_wait(epfd, events, 1024, -1);

    // 只遍历就绪的 nready 个，不是所有 fd
    for (int i = 0; i < nready; i++) {
        int connfd = events[i].data.fd;
        if (sockfd == connfd) {
            int clientfd = accept(...);
            epoll_ctl(epfd, EPOLL_CTL_ADD, clientfd, &ev);
        } else if (events[i].events & EPOLLIN) {
            recv(...); send(...);
        }
    }
}
```

**核心改变：** 注册与等待彻底分离。内核用红黑树存储监听集合，就绪链表存储已就绪的 fd，`epoll_wait` 直接返回就绪列表。

**相比上一代解决了：**
- **O(1) 获取就绪 fd**：`epoll_wait` 只返回就绪的，不扫描全部
- **无需每次拷贝**：fd 注册一次，内核持久维护
- **无 fd 数量上限**：受系统 open files limit 控制
- **支持边缘触发（ET）**：减少系统调用次数，进一步提升性能

**仍有改进空间：**
- 业务逻辑（accept/recv/send）硬编码在主循环的 `if/else` 中
- `if (sockfd == connfd)` 靠条件判断区分连接类型，扩展性差
- 仅 Linux 支持

**适合场景：** Linux 高并发服务器，10 万+ 连接，是现代高性能服务器的标配。

## 第五代：Reactor 模式

**文件：`reactor.c`**

```c
// 用函数指针表替代 if/else
struct conn_item {
    int fd;
    char buffer[BUFFER_LENGTH];
    union { CALLBACK accept_callback; CALLBACK recv_callback; } recv_t;
    CALLBACK send_callback;
};

// 主循环：纯粹的分发，不含任何业务逻辑
while(1) {
    int nready = epoll_wait(epfd, events, 1024, -1);
    for (int i = 0; i < nready; i++) {
        int connfd = events[i].data.fd;
        if (events[i].events & EPOLLIN)
            connlist[connfd].recv_t.recv_callback(connfd);  // 查表调用
        else if (events[i].events & EPOLLOUT)
            connlist[connfd].send_callback(connfd);          // 查表调用
    }
}
```

**核心改变：** 在 epoll 基础上引入**回调机制**，将业务逻辑从主循环中剥离，主循环只做事件分发。

**相比上一代解决了：**
- 主循环不含业务逻辑，只负责"查表→调用"
- 新增事件类型只需绑定新回调，不需改主循环
- fd 类型（listenfd vs clientfd）通过回调指针区分，无 `if (sockfd == connfd)` 硬判断
- 回调内部可动态切换监听事件（EPOLLIN ↔ EPOLLOUT），驱动状态流转

**这一代是设计模式层面的提升，性能与 epoll 相当，但架构更清晰，是所有高性能框架的基础。**

## 五代模型横向对比

| | tcp_server | select | poll | epoll | reactor |
|--|-----------|--------|------|-------|---------|
| **并发方式** | 多线程 | 单线程 | 单线程 | 单线程 | 单线程 |
| **fd 上限** | 受线程数限制 | 1024 | 无限制 | 无限制 | 无限制 |
| **内核扫描** | 无（阻塞等待） | O(n) | O(n) | O(1) | O(1) |
| **用户态遍历** | 无 | O(n) | O(n) | O(k) | O(k) |
| **每次调用拷贝** | 无 | 整个位图 | 整个数组 | 仅就绪事件 | 仅就绪事件 |
| **跨平台** | 好 | 最好 | 较好 | 仅 Linux | 仅 Linux |
| **代码扩展性** | 中 | 差 | 差 | 中 | 好 |
| **适合并发量** | < 千 | < 千 | < 千 | 10万+ | 10万+ |

## 性能瓶颈的演进

```
并发量
  │
10万+│                                        ●  epoll
     │                                        ●  reactor
     │
  万 │
     │
 千  │              ●  poll
     │         ●  select
     │
 百  │  ●  多线程
     │
  1  │  ●  单线程阻塞
     └────────────────────────────────────── 演进时间
```

每一代的出现，都是上一代触及瓶颈后的必然产物：
- **单线程阻塞** → 只能服务一个客户端，催生了多线程
- **多线程** → 线程开销无法支撑高并发，催生了 I/O 多路复用
- **select** → fd 上限和复制开销，催生了 poll
- **poll** → O(n) 扫描无法应对海量连接，催生了 epoll
- **epoll** → 业务逻辑与分发混杂，催生了 Reactor 模式
